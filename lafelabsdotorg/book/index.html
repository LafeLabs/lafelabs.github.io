Introduction

Geometron is two things: a language, and a way of organizing information. This whole project is based on the simple idea that as a civilization we already know far more than we need to and that the next frontier in knowledge is in organizing the knowledge we already have. Another key idea used to build this structure is that the so-called "open web" is the single most powerful information processing system ever devised and that any attempt to better organize information must be based on it. Geometron is also based on the idea that communication based on geometric ideas like symmetry and shape can have enormous power and have been vastly under-used due to the emphasis our civilization has put on numbers.

In some sense, geometron is a computer language. The vast majority of what is found in this volume is based on building a language that exists on computers and was written in computer languages(mostly javascript, html, and css, the languages of the open web). But to lump it together with such languages as C or java would be to miss the point. It is more like a human language in the sense that while you can use a computer language to do things like search for words in a English document or compose a book in English, that does not make English a computer language. Or is it? Without comments, computer code is impossible to maintain or build in teams and without a output in a human language it is usually not very useful. And which is math? Machines do math, but generally are programmed in something more tailored to the machine and the human doing the coding.

This idea of math-as-language really gets to the heart of the matter. Geometron seeks to build a language that is useful both for human-human communication(like English) and for human-machine communication(like javascript), but which is independent of both. That is to say, it should be possible to express things in this system using only shapes and symbols in a self-contained way, then run those operations on a machine or send them to a human without ever writing a line of code in javascript or C or a line of language in English or Chinese. That is, this is an attempt to organize existing geometric knowledge into a language which can be used to communicate.

As I said in the first paragraph here, this language is only half of Geometron. The other half is the way that information is organized. This means doing things like considering how to sort words from images from equations in a way that helps organize document creation.

There is a tradition in the world of computer languages of introducing a language by printing the words "hello, world" in the simplest possible way as an example. I'm going to take the broader view of the "hello world" in this case however. Part of the point of geometron is that it's not based on a human language.

There are three commands here, shown in the graphical code above here. The first and last are the command for "draw circle", and the arrow indicates moving one unit to the right. There are several ways to express this in Geometron. Each of these commands also have an address in a stack of 8X8 grids. To illustrate the power of geometron, I will now construct the most basic shapes beyond the circle: the equilateral triangle, square, regular pentagon and regular hexagon. I'll go into much more detail about how the commands used here work later, but for now I'll just jump in with examples.

The Geometron language is not just a computer language like C or a human language like English. It's more of a hybrid which includes building physical objects, navigating the physical world, creating physical documents with various media as well as communication with both humans and machines. Therefore the structure will be unusual in that it will move freely between these spheres. I will try to provide examples in multiple of these layers.

The Laws of Geometron

Geometron, like most languages, has a set of laws. In human languages these laws often take the form of grammar. In computer languages, they are syntax, and are generally very precise in specifying what is or is not valid code. This is not quite what I mean by the "Laws of Geometron", however. Those more syntax-like rules are documented elsewhere in this book.

The Laws of Geometron are a set of guides for how to think about the world when creating new things using Geometron. This whole project is intended not just as another computer language but as a whole different approach to communication with technology. I believe that some un-examined assumptions about philosophy have caused us(modern civilization) some problems, and that those can be corrected by trying other sets of assumptions. To that end, I will present the laws that I live by as the creator of this language, then explain what each of the laws means and how it affects what we choose to do in using and developing Geometron further in the future.

The Laws of Geometron

  1. Everything is physical
  2. Everything is fractal
  3. Everything is recursive
  4. No money
  5. No mining
  6. No property

EVERYTHING IS FRACTAL. By this I mean that one idea to constantly keep in mind when designing a system with Geometron is that we should always focus on the relevant scale in a given situation. One of the problems I have with modern industrial technology is that everything is set to the maximum possible precision by default. For example, due to the economies of scale of microfabrication, we fabricate microchips in huge batches on wafers that are several 10's of cm across but have precision of all components on the 100 nm level. A Geometron based approach to microfabrication would not have a single precision, because the goals are different on a human level. If I'm fabricating one processor for my own use, I don't care about the same things as Intel or Motorola and view this more as a craft than an "industry" in the modern sense. And as with all crafts, there will be some slop built in based on what scale I'm working at. Thus, I might place the overall work area into position on a cm scale, fix the tool position on a mm scale, scan it with 10 micron precision, and then have a sub-tool affixed to it which can fill in details on a sub micron scale. I might also have another sub-tool affixed to that which can work at the atomic scale, but in all cases the relative position uncertainty is not smaller than 1%.

In software situations, the number of pixels in a field determines a reasonable level of precision in most cases. Thus for a 1000x1000 pixel area, 0.1% is the maximum that it ever makes sense to care about. Realistically this is far more than we usually care about, though, since for example an icon on a desktop might be just a few pixels across, making it have a relative precision locally of a few percent. Also, the exact position of the icon on the desktop relative to the overall screen shape is almost certainly not less than a few percent. While these statements might seem obvious and trivial they are at odds with the behavior of engineers in almost all cases, especially when computers are involved.

A final note on "everything is fractal" must be made on the limits of this law. I find the work of Mandelbrot in his book on the fractal geometry of nature to be pathalogical because it assumes a totally non-physical range of fractals. In reality the world around us is almost always described by fractal structure, but that fractal nature always has limitations. A leaf has veins in it which repeat in their behavior on multiple scales, but only up to a point: below the scale of a few leaf cells vein-like behavior is lost, and while vein like nature may exist in the branches that carry the leaves, it stops at the forrest level, and there is a clear phase change between leaf-vein and branch. Ideas like those of Mandelbrot, like those of Newton mentioned above, can be both powerful and beautiful, but to view them as absolute truths with an existence independent of the human mind, with infinite extent in some imaginary universe is just religious nonsense and is holding back our ability to build more robust technological systems.

EVERYTHING IS RECURSIVE. Fractals and recursion go together like peas and carrots. Recursion is when something makes reference to itself. Human language is by its very nature recursive. We use language to build language, constantly using words that reference words, sentences about sentences, books about books, ideas about ideas and so forth. By stating in black and white that everything is recursive we remind ourselves to always be thinking of this aspect of language and to use it constantly to make it as natural and easy to use as possible. As with the fractal nature of things recursion should be considered to be both everpresent and finite. That is, things can call themselves but they should not do so too many times, creating infinite loops. Infinite loops are non-physical which brings us to the next law:

EVERYTHING IS PHYSICAL.A popular quotation in physics is from Rolf Landauer, which says "information is physical". This is quoted repeatedly in both mesoscopic physics and quantum computation theory. I'm not sure what this is supposed to mean, but I'm skeptical of its intent. What does it mean for a thing to be physical? What is a thing that exists and how do we distinguish it in language from that which does not exist? I'm not sure there are easy answers to these questions. My observation from an anthropological view is that this phrase "information is physical" is used by physicists to denote the type of pontification that they are about to engage in, just as "there be dragons here" is used to denote humorous intent in regards to the incomprehensible in physics theory. In Geometron I use the "law" that everything is physical to simply state a philosophical view that the "non physical" is not a thing that is of use to us. This is no more or no less a proper scientific statement than "information is physical", but it is a position statement for a body of work. Either things are as you see them and feel them and interact with them or they are nothing. Disembodied ones and zeros on an imaginary tape in an imaginary space don't exist. Everything is either real or not. And real things can be interacted with. For the record, this is a position in which string theory is not a science, and is a poor excuse for a religion.

NO MONEY. Money is, like so many things we take for granted, just a model. It has its place and has been very productive in some spheres. However, in software it has had an extremely corrosive effect on everything. Software and money, when combined, always create injustice and evil. I state this without proof. If it is not obvious to the reader our world views are totally incompatible. However, just was we all use knowledge created by people from hundreds of years ago with totally different ideology from our own, it should be possible for those who believe software and money can coexist without doing Evil to use this work. Just don't expect cooperation or collaboration of any kind from the author or anyone working directly with the author. The work directly documented in this first version of this manuscript is released for free into the world, and will be further worked on as much for free as possible. I'm not sure how to make a living and survive in capitalist America this way, but I would rather fail at that than participate willingly in what I see as pure Evil, so this is my choice.

NO PROPERTY. Property is Evil. I will not attempt to argue with the reader on this subject. This is a belief I have come to through hard experience of what I consider to be Evil, with a capital "E". Accept it or not. But for the purpose of this project I reject any form of property, "intellectual" or otherwise. By the power vested in me by the evil "laws" of the propertarians, I release all my work presented here into the public domain with no restrictions whatsoever. I have reached a point in my life where I have decided to fight for what I believe in, to the death if needed, and I believe the claim to own an idea is evil.

NO MINING. The current industrial system consists of the following process: dig things up, use them to make people miserable, set them on fire, and bury them in the ground. It is time to break this cycle. Writing a piece of art software in javascript is hardly a major blow to all this, but it is another law of this project to at least attempt to minimize mining in all cases. That is why the physical implementations of Geometron are all built from trash when possible.

HTML for Geometronians

Most of us are not great writers. And yet we take literacy for granted as a basic skill needed to function in society. Some people have argued that "code is literacy" now in society. On its own this is a very bad idea. Most of the code that professional software developers use changes very fast and is very specialized and building systems that require everyone to master that level of constantly changing highly technical material is unrealistic and simlpy leads to more divisions between the information haves and have-nots.

This book is predecated on a different path than the one in which every person on Earth becomes a software developer and the one where those of us who don't all have our lives controlled by those who do. Just as non-writers can still write an email or a text, non coders should be able to make simple-but-effective bits of code that can get the basic tasks of life done, just without the level of sophistication required by a professional. As the so-called "Internet of Things" expands, as well as the mobile web, the number of potential web servers with documents on them will rapidly exceed the total number of humans on the planet. These hardware devices are approaching zero cost, and thanks to the ubiquity of Cloud computing and internet of things, anyone who can master basic web code can be their own node in a totally decentralized version of the Internet.

I dream of a day when the current centralized Internet controlled by a few billionaires in California is totally replaced by a decentralized model where there are as many web servers as there are people on Earth, and roughly one in ever 100 people is the equivalent of a very basic level web programmer. The way for this to happen is to ignore the trendy tools of the fast-moving commercial Internet and stick with the most open standards which have very wide support both in for profit and non profit areas, as well as totally internaionally. What this all boils down to is that which runs in any web browser is vastly more free than any other kind of software. To make things that run in a web browser we need three things: HTML, CSS, and JavaScript. In this section I guide you into learning HTML.

HTML stands for "Hypertext Markup Language". It is the language which is used by all web browsers to format all the text and graphics we see when we look at a web page. HTML is the universal language of the World Wide Web. As of this writing, in 2017, the most up to date version of HTML is called HTML5, and that is what Geometron uses as a basis for everything we do on computers. In the philosophical system of Geometron, we will try to create as close as possible to unversal literacy in our language for those who participate in it. This means we all need some familiarity with the way that HTML works at least.

What HTML does specifically is turn a text-only format into the vast number of types of thing you can see in your web browser. It can be thought of as a system for mapping just the limited number of keys on your computer keyboard to all the things you've ever seen on the Web: images, videos, drawings, cartoons, games, etc. etc. All these things come from a sequence of characters interpreted a certain way.

The most basic thing to understand in HTML in order to get started is the special role played by the "less than" and "greater than" symbols, or angle brackets: "<" and ">".

Getting Started

There are so many ways to get started with creating computer code that it can be overwhelming. Due to experts constantly strongly disagreeing with each other, tutorials tend to try to stay neutral in these disputes. The problem with this is that it strands the total beginner in a series of random, frustrating web searches. Thus I will somewhat arbitrarily choose and recommend a tool, the one I've decided to use.

I wanted a tool that was free--not just free but really free, as in you don't get annoying reminders constantly begging for money. I also wanted something with a large number of users and a significant effort behind it so that it won't be abandonned. I chose the free code editor from Microsoft, called Visual Studio Code. It can be downloaded for free and installed on pretty much any computer system. Here is the link to get it directly from Microsoft:

https://code.visualstudio.com

As a non professional, I probably use less than one percent of what this software can do, but it's good at staying out of my way for my simple tasks, and it's nice to know that if I do want to do more sophisticated or professional tasks that it can do that as well. So my advice is that if you are like me and you're starting from scratch with no code background, just download and install this, and go from there.

Once you have installed Visual Studio Code, open it, and start a new file using the file menu to select "new" or using command-N or control-N depending on your operating system. Now make a directory somewhere on your computer for your code stuff, and save your new blank file as "index.html"(without the quotes). The ".html" ending is mandatory for your computer to recognize this as HTML rather than some other type of code. It's useful to save this immediately not just to be in the habit of frequent saving but to tell your code editor that this is HTML. That will immediately cause it to use highlighting to make everything much easier to read.

Now copy/paste the following code into the file(upper and lowercase letters are equivalent in HTML): 

<!DOCTYPE HTML>
<html>
<body>
Here is some text.
</body>
</html>

Make sure to save that, then double click the icon for the file to open it in a browser. That's it! That is a file in the format readable by the several billion devices and people connected to the Web. You can email a file like this to someone and they can read it, but that's true of a text document as well and so far we've added little. To feel the true power of the web, this has to get put online somewhere. We will deal with this in more detail in the "server side Geometron" chapter. For this tutorial we will again somewhat arbitrarily choose a single method of doing this and stick with it for now. Part of the idea of this book being all in the public domain is that future users can change this as needed(now that you know how to edit HTML you can do that).

For free and easy web hosting I use https://www.000webhost.com. Get an account, which, again, is free. You'll need to put in your email address. I have had no spam issues, I think this is fine. Once you have an account go to "file manager", then click on "upload files now". You should see a folder called "public_html". Having a folder of this name is fairly universal when building a website. Now drag the file called "index.html" into that folder. Then go back to the tab on your browser that links to your site, which should be a address right under the word "File Manager", which ends with "000webhostapp.com", and starts with whatever name you picked. Click on that link. You should now see your file in your browser as before but now it's on the real Web! The number of pople who can read your file just jumped from one to about 3 billion. Really. This is staggering in its power and is worth contemplating at this time.

Now, go back to your code editor(Microsoft Visual Studio Code) and change the text to something actually interesting. It doesn't matter what! We'll make it look cool soon, but for now just change the words. Now save it, drag and drop the new vesion of index.html over into 000webhost.com(it should ask you if you really want to over-write, and you do).

Some Useful Tools

One of the principles on which Geometron is based is the incredible power of the modern web browser. The major browsers are Chrome(by Google, used on Android Devices but runs on everything), Safari(Apple), Internet Explorer(Microsoft), Mozilla Firefox, and Opera(made by Opera). Everything we do here is supposed to work on all browsers, and mostly it all should. The exception will be when we start working with JavaScript, some things won't run in some browsers until you change the settings to allow scripts. A rule of thumb is also that the worst browser will always be IE. Microsoft has slowly given up on their campaign to break the Web, but it still lingers in IE, and some users have legacy copies of much older versions of IE which very deliberately break everything. Avoid IE if you can. If you're using a computer where IE is the default browser I recommend installng any of the other browsers listed above(shouldn't matter which), otherwise whatever your default browser is is probably fine.

All browsers have, by various names, "developer tools", which are incredibly useful as not just a development but a learning tool. Whatever browser you have, google "[name of browser] view source" and you can find instructions on viewing the source code of any site using your browser. Do this! And repeat. Again and again, all across the Web. This is why it's called the "Open Web", and why profit driven companies like Microsoft and Oracle have worked so hard to break it. The Web is totally open. Without any special tools every single user on the Web can see all the code(most of it at least) of all the billions of sites out there. When you put your stuff on the Web, everyone can read that as well. So not only does this mean you can publish anything instantly to 3 billion people, they can read the equivalent of all your editorial markup for the printer as well, and see inside the workings of any active scripts you write, all without leaving their browser. Incredible.

One thing you'll immediately notice as you start reading web sites' code is how some sites are easy to read and some are hard. Often the more professional sites will have tons of hard to read code. This is usually because those sites are not written by hand like what we are doing, but as part of some huge software system where the computer mostly handles the actual production of code seen the the browser. This is one of the tensions that exists in the structure of the Web. On the one hand, the free and open system allows people like us to make a new site in 10 minutes, which has incredible power. On the other hand, if you want a site viewed by a million people a month, and you have a team of hundreds of developers buildng that site, writing out things by hand like we are doing is not practical. One would hope that in the best interests of the Web that neither one "wins", and that huge systems continue to streamline the process of building things with a thousand developers and a billion users, while the browsers and developer ecosystems all continue to support the process of one amateur building a hand coded site from scratch in a few minutes and sharing it with a dozen people.

Template

With this reference list out of the way and to look back at as needed, I will now create a template which uses all these elements which you can copy and paste into an html file (like index.html). You can then delete what you don't need and copy/paste what you need more of and edit as you see fit to make any simple html content you might want. That being said the page made here will look very simple and plain and will be what is sometimes called "web brutalist" design. Special Characters

The last bits of HTML I want to put in here because they are so widely useful and are not always in the early introductory materials is special characters of varoius kinds. First of all, all the above code, if you look at the source code of this document, deals with the unique problem in html of formatting the "angle brackets" used for all the varoius html tags. As I type this, if I want to express that symbol, and I just hit the key with the symbol on it, I'll see that in my editor, but the browser will always try to turn it into some kind of tag and will fail catastrophically and unpredictably to render your intended text. The workaround is the use of the ampersand: "&", which preceds the letters "lt" for "less than" and "gt" for "greater than". Of course this immediately opens yet another problem: if ampersand is used to mark special weird characters, how do you tell the browser to acually just print an ampersand? To print an ampersand, put "&amp" into your text editor. Note again that if you view the source of this document, that in order to have the browser render "&amp" I had to type "&ampamp". And in order to type that last explanatory sentence I had to type a third "amp"!

Finally we must address how to use unicode. One of the motivations of Geometron is to, eventually, replace unicode in many situations. Unicode is a list of ludicrous length(136,755 characters according to wikipedia) of just about every random symbol or character that people could think of to put in the standard. It has many languages, and can be useful in a pinch for grabbing one character from any given language, but the thing we use it for the most in Geometron is quick access to simple characters that are independent of human and machine language: arrows. Unicode is inserted into html as follows:

&#[decimal code for your unicode here] or &#[hexidecimal code for your unicode here]

The way to find the unicode for something is to google "[thing you are looking for] unicode" and you'll probably find it. Be careful to note if the code you find is hexidecimal(base 16) or decmial. Usually you can spot hexidecimal by letters, which are usually in all caps, for example: "&#x25C0" for "◀"(note the letter "C"). The drawback of unicode goes to the heart of the philosophy as explained in the Laws of Geometron, in this case "everything is fractal". In almost no situations do you want to use 100 languages at the same time, along with a thousand arrows. However, after you choose a langauge, being limited to just 2 or 3 typefaces is really frustrating--you are more likely to want a dozen or so(not for one document necesarilly as that's usually bad design but across your overall work/life). You want huge weird looking art fonts, tiny fonts with nubs on them to make them readable, rainbow colored fonts, super modern designer fonts, etc. And yet if unicode had that level of fractal complexity for every language it would be so big as to simply not function at all. So you end up with, as is so often in computers, a dreary heirarchy of importance that gets imposed on all users. Thus the English-speaking user, no matter how little they need of creative typesetting has a vast sea to choose from, while the user of traditional written Mongolian, a beautiful flowing vertical script is forced into one uninspired font, with no control over it and no options. It is my hope that with Geometron a skilled caligrapher who is an expert on their script in the world of pen and ink but has no computer skills should be able to build numerous scripts on their own without consulting some international font bureaucracy to ask permission to write in their own language on their own hardware.

0: Creator

The "seed" or "creator" element of the hypercube is simply a program to convert a raw hypercube in byte code format into a fully formatted HTML file. This is in part to build files up from physical data tablets, which can be scanned in, converted to bytecode, put into a text file, and then converted. Or it can be used for hypercubes stored on remote servers, to build a full instance locally on a server.

The ASCII encoded information in address 0 gets converted into a Python program called "creator.py", which, when executed, will create index.html. A separate program is used to convert the raw bytecode and make this file is called "metacreator.py", and this should could be written in any language, such as a native shell script. An example of metacreator.py is as follows:

01-05:Base Cube

This set of ASCII bytecode data represent combined HTML, CSS and Javascript, with enough detail so that when converted to text and put in index.html it forms the basic core of the whole document, and should then be readable by a browser.

06-037:Root Magick

These commands, like 0300-0377, are part of the JavaScript function called doTheThing(). This function is discussed in a separate section, as it is the beating heart of the Geometron Virtual Machine(GVM). In general this is where commands go which enable editing of glyphs, such as moving the geometric cursor around during glyph editing, deleting glyphs, moving to different glyphs in the Hypercube, etc. These commands are given the name "Root Magick" to denote their incredible power. They are the commands used as the basis to alter the very fabric of the language, and by using the words "root", denoting a fundamental thing, and "magick", referencing certain types of the concept of deep and mysterious powers, I try to emphasize just how powerful this whole thing is.
